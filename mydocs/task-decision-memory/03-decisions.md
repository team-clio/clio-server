# Decision Memory 결정 히스토리

각 항목: **후보 → 선택 → 이유**. plan의 결정 포인트 D1~D8.
전제: #7·#8 substrate 재사용(`EmbeddingClient` 범용, pgvector `real[]`+캐스팅 패턴, in-memory 코사인 CI 대체,
`IssueEmbedding`/`IssueVectorSearch`/`IssueMemoryService`/`SimilarIssueEntry` 미러링).

D1·D2·D5·D7은 사용자에게 트레이드오프를 설명하고 확정. D3·D4·D6·D8은 #8 선례를 그대로 잇는 것으로
사용자 동의(이견 없음) 하에 추천대로 확정.

---

## D1. 결정 메모 스키마

**후보**
- (a) title + body(최소).
- (b) + category/status.
- (c) + author/date 등 운영 메타.

**선택: (a) title + body(최소)** — createdAt은 표시용 자동 포함.

**이유**
- 이번 목표는 "등록→검색→분석연동" 배선 검증. 두 필드로 충분.
- category/status는 충돌 필터·정렬이 본격화되는 LLM 단계(#10~11)에 실제 쓰임이 생길 때 추가해도 늦지 않음(YAGNI).
- 지금 넣으면 값 채우는 UI/검증만 늘고 소비처가 없어 죽은 필드가 됨.

---

## D2. 등록 범위(CRUD)

**후보**
- (a) 생성 + 프로젝트별 목록 조회.
- (b) + 단건 조회.
- (c) + 수정/삭제(재임베딩).

**선택: (a) 생성 + 목록 조회**

**이유**
- 분석 연동을 증명할 최소치. #8 IssueMemory도 remember/findSimilar 최소로 시작.
- 수정/삭제는 재임베딩 훅만 있으면 쉬우나 흐름 검증엔 불필요. 결정이 틀리면 지우고 재등록.

---

## D3. 결정 임베딩 텍스트 구성

**후보**
- (a) title + body.
- (b) title만.

**선택: (a) title + body**

**이유**
- 결정의 의미는 본문(이유·맥락)에 있음 → body 포함해야 검색 신호가 산다.
- #8 I1의 "쿼리와 대칭" 원칙과 달리 **여기는 의도적 비대칭**: #8은 리포트↔리포트(대칭이 맞음), 여기는
  리포트↔결정이라 애초에 다른 종류 텍스트의 교차 검색. 결정=title+body, 쿼리=리포트 title+desc.
- 비대칭이라 API embedding 전엔 교차 품질이 특히 낮음 — 뼈대 검증 목적임을 result에 명시(D8/리스크).

---

## D4. 쿼리(리포트) 임베딩 소스

**후보**
- (a) 리포트 title + description(#8 `IssueMemoryService.embeddingText`와 동일 규칙).
- (b) + 분석 준비물(candidateDomains·codeSearchTerms).

**선택: (a) 리포트 title + description**

**이유**
- #8과 같은 소스라 규칙 재사용 가능(중복 최소).
- 분석 준비물까지 넣으면 결정 검색이 파이프라인 중간 산출물에 결합돼 순서 의존↑. 뼈대는 단순히.

---

## D5. 결정 벡터검색 구현 — 제네릭화 vs 병렬 (rule-of-three 지점)

**후보**
- (a) `VectorSearch<T>`로 제네릭 추출(코드·이슈·결정 통합, #7·#8도 리팩터).
- (b) 결정 전용 `DecisionVectorSearch` 병렬 구현(#8 패턴 복제).
- (c) 저수준 코사인 primitive만 추출 + 엔티티별 얇은 로더.

**선택: (b) 병렬 구현 — 통합(추출)은 나중으로 미룸**

**이유**
- #8 I4에서 "세 번째 사용자 생기면 (c)로 추출"이라 적었고 지금이 세 번째 → 이 결정은 **그 약속을 지금
  강행하지 않겠다는 명시적 판단**이다.
- 이유: pgvector 네이티브 쿼리에 테이블·컬럼명이 박혀 있어 순수 추출(b→c)이 생각보다 크고, #7·#8까지
  동반 리팩터하면 이번 작업이 비대해진다("작게 쪼개기" 규율 위배). 정작 다른 부분(반환타입·테이블)이
  추출로 안 없어져 이득도 제한적.
- **1차는 복제로 빠르게 미러링하고, 중복이 실제로 아픈지(네 번째 사용자·버그 동조 수정 등) 본 뒤
  (c) 추출을 별도 스텝으로 판단**한다. 사용자 동의 하에 확정.

---

## D6. 분석 파이프라인 연동 지점

**후보**
- (a) `AnalysisWorker` 통합 + 조회 서비스 노출(둘 다).
- (b) 별도 조회 API만.

**선택: (a) 둘 다** — #8 I5와 동일 판단.

**이유**
- 3.3 목표가 "분석 시 관련 결정 검색"이라 워커 통합 필요.
- 조회를 서비스로 두면 UI·MCP가 재사용.
- **연동 형태**: `AnalysisWorker#run`에서 리포트로 관련결정 top-k 조회 → draft/Response에 참고로 포함.
  Response에 `RelatedDecisionEntry` record 신설(`SimilarIssueEntry` 미러). 결정 0건이면 빈 리스트.
- #8과 달리 remember(자동 저장)는 없음 — 결정은 사람이 등록(D2)하므로 워커는 조회만.

---

## D7. 충돌 감지 범위 — 3.3 셋째 목표를 어디까지 (중대)

**후보**
- (a) 관련 결정 검색·표시만(충돌 판정은 사람 몫).
- (b) + 얕은 rule 신호(추천 문구 ↔ 결정 키워드 대비).
- (c) LLM 기반 충돌 판정.

**선택: (a) 관련 결정 표시만** — 자동 충돌 판정은 #11로 미룸.

**이유**
- embedding이 가짜 semantic인 단계라 (b) rule 충돌은 오탐이 많아 신뢰를 깎는다(틀린 경고의 해악).
- (c) LLM 충돌 판정은 3.3이 아니라 로드맵 4.3/4.4(#10~11) "리포트 생성·근거 검증"에 자연히 속한다.
- 3.3 셋째 목표("추천이 기존 결정과 충돌하지 않게")는 **관련 결정을 분석 결과에 보여줘 사람이 충돌을
  알아채게** 하는 것으로 실질 충족. result에 "표시로 만족, 자동판정은 #11" 명시.

**선택: #7·#8과 동일** (대안 없음 — pgvector는 H2에서 못 돎)
