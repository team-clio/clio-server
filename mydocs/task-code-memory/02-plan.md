# Code Memory 구현 계획

overview 컨펌 완료. 이 문서는 **구현 단계**와 **사용자가 결정할 항목(결정 포인트)**을 담는다.
결정 포인트는 3단계에서 하나씩 확정하고 `03-decisions.md`에 기록한다.

## 전체 그림

```
스캔(CodeAnalysisService.scan)
  └─(A) chunk 생성  ── CodeChunk 저장
         └─(B) embedding 생성 ── 벡터 저장
분석/검색
  (C) 쿼리 → 쿼리 embedding → (D) 벡터 유사도 top-k
  (E) 키워드 검색 결과 + 벡터 검색 결과 = 하이브리드
         └─(F) 분석 파이프라인(CodeCandidateRanker) 연동
```

## 구현 단계 (결정에 종속)

각 단계는 관련 결정 포인트가 확정된 뒤 착수한다. 단계마다 컴파일/테스트 통과 상태로 커밋한다.

- **S1. Chunk 모델·저장소** — `memory.CodeChunk`(엔티티) + `CodeChunkRepository`.
  - 필드(안): projectId, fileId, path, symbolName, role, chunkType, startLine, endLine, (본문?), embedding(?)
  - 관련 결정: **D1(granularity), D2(본문 저장), D4(벡터 저장 형태)**
- **S2. Chunk 생성** — 스캔 시 `CodeSymbol` → `CodeChunk` 변환기(`CodeChunker`).
  - 관련 결정: **D1, D2, D7(재생성 시점)**
- **S3. Embedding 경로** — `memory.EmbeddingClient` 인터페이스 + 구현.
  - 관련 결정: **D3(embedding 소스)**
- **S4. 벡터 저장·유사도 검색** — chunk 저장 시 embedding 부착, 쿼리 벡터로 top-k 조회.
  - 관련 결정: **D4(저장·검색 방식)**
- **S5. 하이브리드 검색** — 키워드 결과 + 벡터 결과 결합 → `CodeMemorySearchService`.
  - 관련 결정: **D5(결합 전략)**
- **S6. 분석 파이프라인 연동** — 하이브리드 결과를 후보 랭킹에 반영.
  - 관련 결정: **D6(연동 지점)**
- **S7. Result 정리** — `04-result.md` + roadmap #7 상태 갱신.

> 조회 API(REST) 노출은 D6에서 "별도 API"를 고르면 포함, 아니면 제외(범위 밖).

## 결정 포인트 (하나씩 확정)

### D1. Chunk granularity — 코드를 어느 단위로 자를까
- (a) **메서드 단위**: 메서드/생성자마다 chunk. 클래스는 헤더(선언+필드)만 별도 chunk.
- (b) 클래스 단위: 타입 하나 = chunk 하나(메서드 다 포함).
- (c) 둘 다: 클래스 chunk + 메서드 chunk 모두.
- **AI 추천: (a) 메서드 단위(+클래스 헤더)**. 의미 검색은 메서드 본문 단위가 가장 잘 맞고, `CodeSymbol`에
  이미 메서드 startLine/endLine이 있어 경계가 바로 나온다. (c)는 chunk 수·임베딩 비용이 2배.

### D2. Chunk 본문(코드 텍스트) 저장 여부
- (a) **DB에 본문 저장**: chunk row에 코드 텍스트 컬럼. 검색·표시 시 파일 재읽기 불필요, 스캔 후 파일이 바뀌어도 일관.
- (b) 라인 범위만 저장: 표시할 때 `rootPath`에서 재읽기(현재 `CodeSearchService` 방식).
- **AI 추천: (a) 본문 저장**. 임베딩 입력·스니펫 표시에 그대로 쓰고, 파일 이동/삭제에도 검색이 안 깨진다.
  용량이 부담되면 길이 상한(예: 8KB)만 둔다.

### D3. Embedding 소스 — 가장 성격을 크게 가르는 결정
- (a) 실제 embedding API(OpenAI 등 `/embeddings`)만. `LlmConfig` 재사용해 endpoint 호출.
- (b) 결정적 로컬 대체만(외부 의존 0): 토큰 해시 기반 고정차원 벡터(bag-of-tokens/해싱). 품질은 낮지만 파이프라인·테스트가 외부 없이 선다.
- (c) **인터페이스 + 두 구현(로컬 기본, API 옵션)**: `EmbeddingClient`를 두고, 설정 없으면 로컬,
  `LlmConfig`가 embedding 지원이면 API 사용.
- **AI 추천: (c)**. 뼈대(#7)의 목적은 "하이브리드 검색 구조를 세우는 것". 외부 키 없이도 전체가 돌고,
  실제 임베딩은 config만 주면 켜진다. 테스트는 로컬 구현으로 결정적.

### D4. 벡터 저장·유사도 검색 방식 (H2 테스트 호환 고려)
- (a) **벡터를 배열/JSON 컬럼에 저장 + 앱단 코사인 계산**: chunk 로드 후 자바에서 top-k. Postgres·H2 모두 동작, 의존성 0.
- (b) pgvector 확장: 런타임 성능 좋지만 Postgres 전용 → H2 테스트에서 별도 전략 필요, 확장 설치 필요.
- (c) 외부 벡터 DB: 과함(범위 밖).
- **AI 추천: (a) 앱단 코사인**. 프로젝트당 chunk 수가 수천 규모라 인메모리 top-k로 충분하고,
  테스트가 그대로 H2에서 돈다. 규모가 커지면 이후 (b)로 교체(인터페이스로 격리).

### D5. 하이브리드 결합 전략 — 키워드 + 벡터 결과 합치기
- (a) **정규화 후 가중합**: 두 점수를 0~1로 정규화, `w_kw·kw + w_vec·vec`로 합산(가중치는 상수).
- (b) RRF(rank fusion): 각 랭킹 순위의 역수 합. 점수 스케일 무시, 튜닝 적음.
- (c) 단순 union 후 기존 ranker 재점수.
- **AI 추천: (a) 가중합**(초기 가중치 예: kw 0.5 / vec 0.5, 상수로 두고 backlog에 튜닝 기록).
  근거가 명시적이라 rationale에 "왜 이 후보인지" 쓰기 쉽다. RRF는 후보군이 커지면 대안으로.

### D6. 분석 파이프라인 연동 지점
- (a) **`CodeCandidateRanker` 입력에 벡터 후보 추가**: 하이브리드 결과를 후보 소스로 합류시켜 기존 랭킹/흐름추적/draft를 그대로 재사용.
- (b) 별도 조회 API(REST)로만 노출, 분석 파이프라인은 이번엔 안 건드림.
- (c) 둘 다.
- **AI 추천: (a)**. #7의 목표("의미적으로 가까운 코드를 찾는다")가 분석 결과에 실제로 반영돼야 의미가 있다.
  기존 키워드 경로는 유지하고 벡터 후보를 **추가**(대체 아님)한다. REST 노출은 필요해지면 후속.

### D7. Chunk·embedding 재생성 시점
- (a) **스캔 시 전체 재생성**: `scan()`이 파일/심볼 지우고 다시 만들 때 chunk/embedding도 같이. 단순·일관.
- (b) 별도 트리거(`POST .../embeddings`): 스캔과 분리, 임베딩 비용을 명시적으로 통제.
- **AI 추천: (a) 스캔 시 전체 재생성**(초기). 단, D3에서 실제 API를 쓰면 스캔이 느려지므로,
  그 경우 (b)를 옵션으로 열어둘지 D3 확정 후 재검토.

## 결정 포인트 요약표

| ID | 주제 | AI 추천 |
|----|------|---------|
| D1 | Chunk granularity | 메서드 단위(+클래스 헤더) |
| D2 | 본문 저장 여부 | DB에 본문 저장(길이 상한) |
| D3 | Embedding 소스 | 인터페이스+로컬 기본/API 옵션 |
| D4 | 벡터 저장·검색 | 배열 저장 + 앱단 코사인 |
| D5 | 하이브리드 결합 | 정규화 가중합(0.5/0.5) |
| D6 | 파이프라인 연동 | ranker 입력에 벡터 후보 추가 |
| D7 | 재생성 시점 | 스캔 시 전체 재생성 |

## 열린 질문 / 리스크

- D3에서 실제 API를 켤 때 스캔 지연·비용 → D7 재검토 트리거.
- 앱단 코사인(D4-a)의 프로젝트 규모 상한 → 커지면 pgvector 이관(인터페이스로 격리해 교체 쉽게).
- 로컬 embedding(D3)의 의미 검색 품질은 낮음 → "구조 검증"이 목적이고 품질 튜닝은 backlog.

## 완료 기준(이 작업)

- 스캔하면 chunk와 embedding이 생성된다.
- 쿼리로 **키워드+벡터 하이브리드** top-k를 얻는다.
- (D6-a면) 분석 결과 후보에 벡터 기반 후보가 실제로 반영된다.
- 외부 키 없이 전체 파이프라인과 테스트가 H2에서 통과한다.
