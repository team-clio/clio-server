# Issue Memory 구현 계획

overview 컨펌 완료. 이 문서는 **구현 단계**와 **결정 포인트(I1~I7)**를 담는다.
결정은 3단계에서 하나씩 확정하고 `03-decisions.md`에 기록한다.

> **#7 substrate 재사용 전제**: `EmbeddingClient`(임의 텍스트 embed)는 그대로 사용. pgvector 패턴
> (`real[]` 컬럼 + 쿼리 시 `::vector` 캐스팅, D4-1)과 in-memory 코사인(CI/dev 대체) 구조도 재사용.
> embedding은 API 켜기 전까진 로컬=가짜 semantic(한계 상속).

## 전체 그림

```
분석 완료(AnalysisWorker)
  └─(A) 이 리포트를 임베딩 ── IssueEmbedding 저장
새 리포트 분석 시
  (B) 새 리포트(title+desc) 임베딩 → (C) 유사 이슈 top-k (자기/같은 리포트 제외)
         └─(D) 분석 결과에 "유사 과거 이슈" 참고로 노출
```

## 구현 단계 (결정에 종속)

각 단계는 관련 결정이 확정된 뒤 착수. 단계마다 컴파일/테스트 통과 상태로 커밋.

- **S1. 저장소** — `memory.IssueEmbedding`(엔티티) + `IssueEmbeddingRepository`.
  - 필드(안): projectId, reportId, (표시용 title/요약?), embedding(`float[]`), createdAt
  - 관련 결정: **I3(저장소 형태), I1(임베딩 텍스트)**
- **S2. 임베딩 생성·시점** — 리포트→임베딩 텍스트 조립 후 `EmbeddingClient.embed`, 저장.
  - 관련 결정: **I1, I2(시점)**
- **S3. 이슈 벡터검색** — pgvector(프로덕션) + in-memory(CI) top-k.
  - 관련 결정: **I4(제네릭화 vs 병렬), I7(테스트)**
- **S4. 유사 이슈 조회 서비스** — 새 리포트 임베딩 → top-k, 자기/같은 리포트 제외.
  - 관련 결정: **I4**
- **S5. 분석 파이프라인 연동** — `AnalysisWorker`: 완료 시 임베딩(A) + 분석 시 유사이슈 조회(D).
  - 관련 결정: **I2, I5(연동 지점), I6(학습 루프 범위)**
- **S6. Result** — `04-result.md` + roadmap #8 상태 갱신.

## 결정 포인트 (하나씩 확정)

### I1. 임베딩 텍스트 구성 — 무엇을 임베딩할까
- (a) **title + description만**.
- (b) title + description + 분석결과(summary·domains·issueType).
- **AI 추천: (a) title+description만.** 쿼리(새 리포트)는 분석 전이라 title+desc뿐 → 저장분도 같은 필드로
  임베딩해야 **유사도가 대칭(apples-to-apples)**. 분석결과는 임베딩에 넣지 말고 **표시용 메타**로 붙여
  "이 유사 이슈는 이렇게 분석됐다"를 보여준다(값은 유지, 대칭성은 안 깨짐).

### I2. 임베딩 시점 — 언제 저장할까
- (a) 리포트 등록 시: 즉시 검색 가능하나 아직 분석결과 없음.
- (b) **분석 완료 시**: 메모리 = "분석된 이슈"만. 유사이슈에 분석결과(요약·추천)를 함께 보여줄 수 있음.
- (c) 별도 트리거.
- **AI 추천: (b) 분석 완료 시.** 유사 이슈의 가치는 "그때 어떻게 분석/해결됐나"에 있음 → 분석된 것만 기억.
  로컬 embedding은 싸서 완료 시 임베딩 부담 없음(API 켜면 D7류 재검토).

### I3. 저장소 형태
- (a) **별도 `IssueEmbedding` 엔티티**(reportId 참조 + embedding + 표시 메타).
- (b) 기존 `BugReport`/`AnalysisJob`에 embedding 컬럼 추가.
- **AI 추천: (a) 별도 엔티티.** 리포트/잡 엔티티를 memory 관심사로 오염시키지 않음. #7 `CodeChunk`와
  같은 결(memory 패키지). 재임베딩·삭제도 독립적.

### I4. 이슈 벡터검색 구현 — 제네릭화 vs 병렬
- (a) `CodeChunkVectorSearch`를 제네릭 `VectorSearch<T>`로 승격해 공유.
- (b) **리포트 전용 `IssueVectorSearch` 병렬 구현**(같은 pgvector 패턴 복제).
- (c) 공통 저수준 primitive만 추출(네이티브 코사인 쿼리 → (id, score)) + 엔티티별 얇은 로더.
- **AI 추천: (b) 병렬 구현(+ (c)는 rule-of-three 시).** 반환 타입이 엔티티별로 달라 제네릭화는 지금 과설계.
  두 번째 사용자일 뿐이라 복제가 싸다. 세 번째 사용자 생기면 (c)로 추출. D0-B의 "범용"은 `EmbeddingClient`
  까지가 진짜 범용이고 store/search는 엔티티별인 걸 인정.

### I5. 분석 파이프라인 연동 지점
- (a) `AnalysisWorker`에 통합: 완료 시 임베딩 + 분석 시 유사이슈 조회해 결과에 포함.
- (b) 별도 조회 API만(분석 결과 미변경).
- (c) 둘 다(워커 통합 + 조회 서비스 노출).
- **AI 추천: (c) 둘 다.** #8 목표가 "분석 결과에 유사이슈 포함"이라 워커 통합 필요(a). 조회는 서비스로
  두면 향후 UI·MCP·#9가 재사용(b). #7 D6와 같은 판단.

### I6. 학습 루프 범위 — "쓸수록 좋아지는"을 이번에 어디까지 (중대)
- (a) **유사이슈 검색만**(retrieval only).
- (b) + 최소 outcome 캡처(분석 후보가 실제 맞았는지 표시하는 훅만).
- (c) 학습 루프까지(피드백으로 retrieval 개선).
- **AI 추천: (a) 검색만.** roadmap 3.2는 "유사 이슈 찾아 참고"까지다. 학습 루프는 outcome ground-truth가
  선행돼야 하고(그 신호 설계가 큰 일), 지금 섞으면 #8이 비대해져 "작게 쪼개기" 규율을 깬다. outcome/학습은
  이후 별도 스텝으로. (오프라인 eval은 이미 `analysis/eval` 하네스가 있음 — 온라인 학습과는 별개.)

### I7. 테스트 전략
- **AI 추천: #7과 동일.** pgvector 실경로는 CI 제외(H2 불가) → 별도 벤치마크. 로컬 embedding + in-memory
  코사인으로 배선·기능 검증(등록→임베딩→유사이슈 조회, 자기 제외). 전체 스위트 H2 그린 유지.

## 완료 기준(이 작업)

- 분석 완료 시 리포트가 임베딩·저장된다.
- 새 리포트 분석 시 유사 과거 이슈 top-k가 결과에 참고로 붙는다(자기/같은 리포트 제외).
- CI(H2)는 배선·기능을 검증(pgvector는 stub/벤치마크). 로컬 embedding으로 외부 키 없이 그린.

## 열린 질문 / 리스크

- embedding이 로컬=가짜라 유사도 품질 낮음(#7 D3-1 켜야 진짜). "구조 검증"이 목적.
- I2(완료 시)면 분석 안 된 과거 리포트는 메모리에 없음 — 의도된 선택(분석된 것만 기억).
- 대칭성(I1-a) 유지: 저장·쿼리 임베딩 텍스트가 같아야 함.
