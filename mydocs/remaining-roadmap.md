# Clio 개발 로드맵

## 목표

Clio의 목표는 사용자가 버그 리포트를 등록하면, 로컬 코드베이스를 분석해서 다음 정보를 빠르게 보여주는 것이다.

- 어떤 코드가 관련되어 있는지
- 수정 난이도가 어느 정도인지
- 변경 위험도가 높은지
- 중요도가 높은 이슈인지
- 어떤 방향으로 수정하고 테스트해야 하는지

지금은 rule-based MVP가 구현된 상태다. 앞으로는 기능 품질을 높이고, 이후 RAG, LLM, LangGraph, MCP 순서로 고도화한다.

## 1. MVP 기능 보강

### 1.1 분석 결과 구조 개선

현재 분석 결과는 빠른 MVP를 위해 단순하게 저장되어 있다. UI나 RAG로 확장하려면 결과를 더 구조화해야 한다.

해야 할 일:

- 관련 코드 목록을 개별 항목으로 다룰 수 있게 정리한다.
- 점수별 판단 근거를 개별 항목으로 다룰 수 있게 정리한다.
- 분석 결과 응답을 UI가 사용하기 쉬운 형태로 바꾼다.

완료 기준:

- 분석 결과 화면에서 관련 코드, 점수, 근거, 추천 내용을 각각 분리해서 보여줄 수 있어야 한다.

### 1.2 관련 테스트 코드 탐색 개선

현재는 관련 테스트 코드 탐색이 약하다. 위험도와 난이도를 판단하려면 테스트 존재 여부가 중요하다.

해야 할 일:

- 관련 클래스에 대응되는 테스트 파일을 찾는다.
- 테스트 파일이 실제로 대상 클래스를 참조하는지 확인한다.
- 관련 테스트가 없으면 위험도와 난이도에 반영한다.

완료 기준:

- 분석 결과에서 “관련 테스트 있음/없음”을 신뢰할 수 있게 보여줘야 한다.

### 1.3 코드 흐름 추적

단순 키워드 검색만으로는 영향 범위를 설명하기 어렵다. Controller, Service, Repository 사이의 흐름을 어느 정도 추적해야 한다.

해야 할 일:

- 클래스 간 의존 관계를 추출한다.
- 메서드 호출 관계를 가볍게 추적한다.
- 관련 코드 후보를 흐름 단위로 묶는다.

예상 결과:

```text
PaymentController
-> PaymentService
-> PaymentRepository
```

완료 기준:

- 분석 결과에서 “어떤 흐름이 영향받는지”를 설명할 수 있어야 한다.

### 1.4 점수화 규칙 개선

현재 점수화는 MVP용 단순 규칙이다. 실제 사용성을 위해 점수 근거를 더 명확하게 만들어야 한다.

반영할 요소:

- 관련 파일 수
- 관련 도메인 수
- 테스트 코드 존재 여부
- Entity 변경 가능성
- Repository 변경 가능성
- 외부 API 관련 여부
- 트랜잭션 관련 여부
- 공통 모듈 관련 여부
- 인증, 결제, 주문 같은 핵심 도메인 여부

완료 기준:

- 중요도, 난이도, 위험도 점수가 각각 왜 그렇게 나왔는지 설명 가능해야 한다.

### 1.5 검색 품질 개선

현재 검색은 MVP 수준이다. 관련 코드 후보의 품질을 올려야 분석 결과도 좋아진다.

해야 할 일:

- 여러 키워드를 동시에 검색한다.
- 정확히 일치하는 결과에 더 높은 점수를 준다.
- 파일 경로, 클래스 역할, 테스트 코드 여부를 점수에 반영한다.
- 중복 결과를 더 잘 제거한다.
- 검색 결과 제한과 정렬 기준을 정리한다.

완료 기준:

- 버그 리포트 키워드로 검색했을 때 상위 결과가 실제 관련 코드에 가까워야 한다.

## 2. 최소 Web UI

백엔드 MVP가 안정되면 간단한 UI를 만든다. 처음부터 예쁘게 만들 필요는 없고, 핵심 흐름을 보여주는 것이 우선이다.

필요한 화면:

- 프로젝트 목록
- 프로젝트 등록
- 프로젝트 상세
- 코드 스캔 실행
- 버그 리포트 목록
- 버그 리포트 등록
- 분석 실행
- 분석 진행 상태
- 분석 결과 상세
- 관련 코드 미리보기

완료 기준:

- 사용자가 브라우저에서 프로젝트를 등록하고, 버그 리포트를 넣고, 분석 결과를 확인할 수 있어야 한다.

## 3. RAG / Memory 고도화

MVP가 실제 프로젝트에서 어느 정도 동작하는 것이 확인되면 RAG를 추가한다.

### 3.1 Code Memory  — 뼈대 구현 완료 (S1~S6, `mydocs/task-code-memory/`)

코드를 chunk 단위로 저장하고 검색할 수 있게 만든다.

해야 할 일:

- [x] 클래스/메서드 단위로 코드를 나눈다. (메서드 본문 + 클래스 헤더, `CodeChunker`)
- [x] 코드 chunk를 저장한다. (`CodeChunk`, 본문 DB 저장)
- [x] chunk별 embedding을 만든다. (`EmbeddingClient`: 로컬 기본/API 옵션)
- [x] 키워드 검색과 벡터 검색을 함께 사용한다. (ranker semantic **fallback** — "필요할 때만 호출")

목표:

- 버그 리포트와 의미적으로 가까운 코드를 찾을 수 있어야 한다.
  → **뼈대는 섬.** 단, 실제 의미검색은 API embedding을 켜야 유효(로컬은 배선 검증용). pgvector 실경로는
  별도 벤치마크로 검증(미작성). 상세·한계는 `task-code-memory/04-result.md`.

남은 backlog: pgvector 벤치마크 러너, API embedding 활성화(D3-1), ANN 인덱스, 임계 튜닝.

### 3.2 Issue Memory  — 뼈대 구현 완료 (S1~S5, `mydocs/task-issue-memory/`)

과거 버그 리포트와 분석 결과를 기억하게 만든다.

해야 할 일:

- [x] 과거 리포트를 검색 가능하게 저장한다. (분석 완료 시 임베딩, `IssueEmbedding`)
- [x] 유사한 과거 이슈를 찾는다. (`IssueMemoryService.findSimilar`, 자기 제외)
- [x] 분석 시 과거 이슈를 참고한다. (`AnalysisJobResponse.similarIssues`)

목표:

- “이전에 비슷한 문제가 있었는지”를 분석 결과에 포함할 수 있어야 한다.
  → **뼈대는 섬.** 단, 실제 의미검색은 #7 D3-1(API embedding) 켜야 유효(로컬은 토큰겹침). pgvector 실경로는
  별도 벤치마크로 검증(미작성). 학습 루프는 범위 밖(I6). 상세·한계는 `task-issue-memory/04-result.md`.

### 3.3 Decision Memory  — 뼈대 구현 완료 (S1~S5, `mydocs/task-decision-memory/`)

설계 결정이나 운영 메모를 분석에 활용한다.

해야 할 일:

- [x] 설계 결정 메모를 등록할 수 있게 한다. (`DecisionMemoryController` POST/GET, `DecisionMemory`)
- [x] 분석 시 관련 설계 결정을 검색한다. (`DecisionMemoryService.findRelevant`, `AnalysisJobResponse.relatedDecisions`)
- [🔶] 추천 수정 방향이 기존 결정과 충돌하지 않게 한다. (D7: 관련 결정을 **표시**해 사람이 충돌을 알아채게 —
  자동 충돌 판정은 LLM 근거검증 #11로 미룸)

목표:

- 코드만 보고 판단하기 어려운 설계 맥락을 분석에 반영한다.
  → **뼈대는 섬.** 단, 실제 의미검색은 #7 D3-1(API embedding) 켜야 유효(로컬은 토큰겹침). 결정↔리포트
  비대칭 임베딩이라 교차 품질은 특히 낮음. 충돌 자동판정은 #11 몫(D7). 상세·한계는 `task-decision-memory/04-result.md`.

## 4. LLM 적용

LLM은 처음부터 판단을 전부 맡기지 않는다. 검색과 점수화는 기존 로직이 담당하고, LLM은 설명과 요약을 담당하게 한다.

### 4.1 버그 리포트 구조화

해야 할 일:

- 리포트에서 키워드, 도메인, 이슈 타입을 추출하게 한다.
- 결과는 JSON으로 받는다.
- 실패하면 rule-based 결과를 사용한다.

### 4.2 관련 코드 요약  — ⏭️ 미룸 (#10에서 명시적으로 제외, `task-llm-report` L1)

해야 할 일:

- 검색된 코드만 LLM에 제공한다.
- 관련 코드가 어떤 역할을 하는지 요약한다.
- 파일명과 라인 번호 근거를 함께 유지한다.

> #10(4.3) 완료 후 별도 스텝으로 진행. 같은 `LlmReportWriter`에 codeSummaries 필드를 추가하는 방식.

### 4.3 분석 리포트 생성  — 뼈대 구현 완료 (S1~S4, `mydocs/task-llm-report/`)

해야 할 일:

- [x] rule-based 점수와 근거를 LLM에 전달한다. (`LlmReportWriter`, grounding 입력 한정)
- [x] 사람이 읽기 좋은 분석 요약을 생성한다. (`GeneratedReport.summary`, 한국어)
- [x] 추천 수정 방향과 테스트 전략을 생성한다. (`recommendedFix`·`recommendedTests`)

> LLM은 3필드 텍스트만 생성, 점수·근거·검색결과는 rule-based 유지(원칙). 실패·미설정 시 rule-based 자동 폴백.
> 환각 방지는 예방(입력 한정+지시)까지 — 사후 검증은 4.4(#11). 상세·한계는 `task-llm-report/04-result.md`.

### 4.4 근거 검증  — 뼈대 구현 완료 (S1~S3, `mydocs/task-evidence-check/`)

해야 할 일:

- [x] LLM이 검색 결과에 없는 파일이나 클래스를 언급하지 못하게 한다. (`ReportEvidenceVerifier`, 사후 검증)
- [x] 근거가 부족한 내용은 제거하거나 경고한다. (E3: **경고**만 — `evidenceWarnings`로 노출, 텍스트는 유지)

완료 기준:

- LLM 결과가 항상 검색된 코드 근거에 묶여 있어야 한다.
  → 근거 밖 언급을 사후 검증해 경고로 가시화(참조 검증). **한계**: 보수적 추출이라 교묘한 환각은 미탐 가능
  (명백한 환각만 잡는 1차 방어). 의미 검증·자동 제거는 범위 밖. 상세는 `task-evidence-check/04-result.md`.

## 5. LangGraph 적용  — 뼈대 구현 완료 (S1~S5, `mydocs/task-analysis-graph/`)

LLM 단계가 안정되면 LangGraph로 분석 절차를 명확하게 나눈다.
(Java/Spring이라 **langgraph4j**로 구현 — LangGraph의 Java 포팅. core 1.5.14.)

흐름(구현된 노드):

```text
prepare(리포트 구조화) -> search(관련 코드 검색) -> flow(영향 흐름)
-> memory(#8 유사이슈+#9 관련결정) -> draft(점수화) -> report(#10 리포트 생성+#11 근거 검증)
```

목표:

- [x] 분석 단계를 분리해서 디버깅하기 쉽게 만든다. (`AnalysisGraph` StateGraph, 노드 6개)
- [x] 각 단계의 입력과 출력을 저장한다. (MemorySaver 체크포인트, state 직렬화)
- [x] 실패한 단계만 재시도할 수 있게 만든다. (threadId=jobId, `invoke(null)`로 재개)

> **한계**: MemorySaver=in-memory라 앱 재시작 후 재개 불가(영속은 postgres-saver 베타뿐 → backlog).
> 선형 그래프뿐(순환·에이전트형은 rule-based 제거와 함께 이후). 동작 보존 리팩터. 상세는 `task-analysis-graph/04-result.md`.

## 6. MCP 적용

MCP는 마지막 단계에 붙인다. 내부 검색과 분석 기능이 안정된 뒤 외부 Agent가 사용할 수 있는 도구 인터페이스로 노출한다.

만들 Tool:

- `search_code`
- `read_file`
- `get_project_structure`
- `find_related_tests`
- `get_analysis_result`

필수 규칙:

- Agent가 파일 시스템에 직접 접근하지 않게 한다.
- 등록된 프로젝트 경로 밖은 읽지 못하게 한다.
- 파일 읽기 범위를 제한한다.
- 호출 로그를 남긴다.

완료 기준:

- Agent가 Clio의 MCP Tool만 사용해서 안전하게 코드 검색과 분석 결과 조회를 할 수 있어야 한다.
