# LLM 근거 검증 구현 계획

overview 컨펌 완료. 이 문서는 **구현 단계**와 **결정 포인트(E1~E7)**를 담는다.
결정은 3단계에서 하나씩 확정하고 `03-decisions.md`에 기록한다.

> **참조 검증(reference check)이지 의미 검증이 아니다.** "언급한 파일/클래스가 근거에 존재하나"만 본다.
> 최대 리스크는 **오탐**(정상 표현을 위반으로 오인) — 추출은 보수적으로, 매칭은 관대하게.

## 전체 그림

```
enrichWithLlmReport (write 성공 시)
  ├ 허용 근거 집합 = relatedCode(파일명·클래스명) + flows(className·filePath)
  ├ GeneratedReport 3필드 텍스트에서 "코드 언급"(*.java, CamelCase) 추출
  ├ 허용에 없는 언급 = 미근거
  └ 위반 있으면 → (액션: E3) / 없으면 → 그대로 적용
```

## 구현 단계 (결정에 종속)

각 단계는 관련 결정 확정 뒤 착수. 단계마다 컴파일/테스트 통과 상태로 커밋.

- **S1. 검증기** — `ReportEvidenceVerifier`: (GeneratedReport, 허용 근거) → 위반 목록/판정.
  - 허용 집합 구성(E1), 언급 추출 규칙(E2), 대상 필드(E5).
- **S2. 액션 + 연동** — `enrichWithLlmReport`에서 write 후 검증 → 위반 시 액션(E3), 노출(E4).
- **S3. 테스트** — 근거 있는 언급 통과 / 없는 언급 탐지 / 오탐 회피 / 워커 폴백.
  - 관련 결정: **E7**
- **S4. Result** — `04-result.md` + roadmap #11 상태 갱신.

## 결정 포인트 (하나씩 확정)

### E1. 허용 근거 집합 — 무엇을 "언급해도 되는 것"으로 볼까
- (a) relatedCode의 filePath(+basename)·symbolName만.
- (b) **위 + flows의 className·filePath.**
- (c) 위 + 도메인·검색어.
- **AI 추천: (b).** flows의 클래스도 #10에서 grounding 입력으로 들어갔으니 언급이 정당하다 → 허용에 넣어야
  오탐이 준다. 도메인·검색어(c)는 코드 식별자가 아니라 굳이 불필요. 허용 집합은 **넉넉히**(오탐↓ 방향).

### E2. 언급 추출 규칙 — 텍스트에서 무엇을 "코드 언급"으로 뽑을까 (오탐 최소화 핵심)
- (a) 공격적: 대문자 시작 단어·경로·점 포함 토큰 다 추출.
- (b) **보수적: (i) `*.java` 파일명 + (ii) multi-hump CamelCase(`PaymentService`처럼 대문자 언덕 2개+)만.**
- **AI 추천: (b) 보수적.** 단일 대문자 단어(`Payment`)·소문자는 추출 안 함 — 한글 문장에서 확실한 코드
  식별자만 잡아 오탐을 최소화. 매칭은 **대소문자 무시·부분일치 허용**(애매하면 통과시키는 쪽). "덜 잡고
  통과"가 "많이 잡고 오탐"보다 안전(신뢰 우선).

### E3. 위반 시 액션 (중대)
- (a) 경고만: LLM 텍스트 유지 + 위반 목록 표시.
- (b) 문장·필드 제거(redact): 위반 문장만 도려냄.
- (c) **전체 rule-based 폴백**: 위반 있으면 LLM 리포트 통째 버리고 #10 rule-based 텍스트 사용.
- **AI 추천: (c) 전체 폴백.** 이유: ①완료 기준이 "LLM 결과가 **항상** 근거에 묶임"이라 위반분 제거가 가장
  충실. ②#10 L3 폴백 기계를 **그대로 재사용**(일관·간단). ③redact(b)는 문장 도려내면 리포트가 너덜너덜.
  ④환각을 한 번 한 리포트는 나머지도 신뢰 낮음. 절충: 보수적 추출(E2-b)이라 폴백 오발동은 드묾.
  → **다만 이건 "좋은 리포트도 토큰 하나 때문에 버릴 수 있다"는 대가**가 있어 사용자와 상의(핵심 결정).

### E4. 경고 노출 방법
- (a) **로그만**(#10 L3 폴백과 동일 방식) — 응답 스키마 불변.
- (b) 응답에 warnings 필드 신설.
- (c) rationale에 append.
- **AI 추천: (a) 로그만.** E3-c(전체 폴백)를 택하면 "제거"가 곧 액션이라 별도 경고 필드가 없어도 roadmap
  "제거하거나 경고"를 충족(제거 쪽). #10 L3과 일관되게 스키마 최소 변경. (E3에서 경고형(a)을 고르면 노출 필요 → 재검토.)

### E5. 검증 대상 필드
- **AI 추천: summary·recommendedFix·recommendedTests 3필드 전부**(합쳐서 한 번에 추출·대조). 셋 다 LLM 생성분.

### E6. 연동 지점
- **AI 추천: 별도 `ReportEvidenceVerifier` 컴포넌트** + `enrichWithLlmReport`에서 write 성공 후 호출.
  검증기는 순수 함수라 단위테스트 쉬움. 검증 자체도 예외 안전(오류 시 통과시켜 분석 안 깨지게).

### E7. 테스트 전략
- **AI 추천**: 검증기 단위테스트 — (1) 근거 안 클래스/파일 언급→통과, (2) 근거 밖 `OrderValidator`·`X.java`→위반
  탐지, (3) 단일 대문자 단어·일반 한글→오탐 안 남(보수성), (4) 빈 근거·빈 텍스트 엣지. + 워커: 위반 시 rule-based
  폴백. 외부 호출 없음.

## 완료 기준(이 작업)

- LLM 리포트가 근거(relatedCode·flows)에 없는 파일/클래스를 언급하면 **탐지되어 제거(폴백)**된다.
- 근거 안에서만 말하는 리포트는 그대로 통과. 정상 한글 표현은 오탐 없이 통과(보수적 추출).
- 검증 로직 오류에도 분석은 안 깨짐(안전 기본값). 전체 스위트 그린(외부 호출 없음).

## 열린 질문 / 리스크

- **오탐 vs 미탐 트레이드오프**: 보수적 추출은 미탐(진짜 환각을 놓침)을 늘린다. 이번은 "오탐으로 신뢰
  깎느니 미탐 감수"를 택함 — 완벽한 환각 차단이 아니라 **명백한 환각만 잡는 1차 방어**임을 result에 명시.
- E3-c 폴백은 좋은 리포트도 버릴 수 있음(오탐 시). 보수적 추출로 완화하나 0은 아님.
- 정밀 추출(AST·NER)은 범위 밖 — 휴리스틱으로 시작.
