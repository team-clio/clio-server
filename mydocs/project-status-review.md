# Clio 진행 현황 & 논의 안건 (2026-07-13)

> 대상: 프로젝트 목적은 알지만 진행 상황은 모르는 사람.
> 목적: 현재 어디까지 왔는지, 일부러 남겨둔 결정이 무엇인지, 오늘 무엇을 논의할지 공유.

---

## 0. 30초 요약

버그 리포트 → 코드베이스 분석 결과를 내는 엔진을 **rule-based → RAG/Memory → LLM → LangGraph** 4계층으로
쌓아 올렸다. **핵심 골격은 모두 동작**한다. 다만 여러 기능이 **"뼈대(배선·구조 검증)까지"** 완료 상태이고,
"실제로 똑똑해지는" 마지막 스위치(실 의미검색·에이전트화 등)는 **의도적으로 열어두고 논의 대상**으로 남겨뒀다.

## 1. 한눈에 보는 진행 현황

| 5 | 검색 품질(랭킹) | ✅ | 리포트 키워드 기반 관련 코드 랭킹 |

## 2. 계층별로 "무엇이 되고, 무엇이 아직인가"

**① 코드 분석(rule-based) — 가장 탄탄함**
- 됨: 코드 파싱·인덱싱, 관련 코드 랭킹, 영향 흐름 추적, 결과 구조화, 랭킹 평가 하네스.
- 아직: 점수화 규칙이 일부 신호(외부 API·트랜잭션·공통 모듈 등)를 반영 못 함.

**② RAG / 3종 Memory — 구조는 완성, "의미"는 아직**
- 됨: 코드·이슈·결정을 임베딩해 pgvector로 검색하는 파이프라인 전부 배선.
- 아직(**핵심**): 지금은 **로컬 임베딩(단순 토큰 겹침)** 이라 진짜 의미검색이 아님. API 임베딩을 켜야 실효.
- 아직: pgvector 실경로는 CI에서 H2로 대체 — 실측 벤치마크 미작성.

**③ LLM 적용 — 신중하게 경계를 둠**
- 됨: 리포트 구조화(JSON+검증), 리포트 생성(요약·수정·테스트), 실패 시 rule-based 자동 폴백.
- 됨: 환각 방어 — 프롬프트 그라운딩 + 근거 밖 언급 사후 검증(경고).
- 아직: 근거 검증이 **경고만**(자동 제거·의미 수준 검증은 안 함), "관련 코드 요약"은 의도적으로 미룸.

**④ 오케스트레이션(LangGraph) — 골격 완성, 확장 여지**
- 됨: 파이프라인을 6노드 그래프로, 체크포인트로 단계 저장·**실패 단계 재개**.
- 아직: 체크포인트가 **메모리 내(in-memory)** 라 앱 재시작 후 재개 불가(영속 라이브러리가 베타뿐). 그래프가
  아직 **선형**(순환·조건 분기 없음).

## 3. 일부러 남겨둔 결정 / 더 고민해야 할 부분 (Open Questions)

> 다음은 "못 한 것"이 아니라 **"의식적으로 열어둔 결정"** 이다. 방향에 따라 크게 달라지므로 논의가 필요.

**A. (전략) rule-based를 걷어내고 풀 LLM/에이전트형으로 갈 것인가?**
- 현재: 검색·점수화는 결정론적, LLM은 설명만. 향후 rule-based를 LLM 판단으로 대체하는 방향을 고려 중.
- 열린 점: 언제·어디까지 전환? #12 그래프 골격은 이를 받을 준비가 됨(순환·조건 분기 추가 가능). 가장 큰 방향 결정.

**B. (실효성) 실제 의미검색을 켤 것인가? (API 임베딩 활성화)**
- 지금 Memory 3종은 로컬 임베딩이라 "구조는 돌지만 의미적으로는 약함". API 임베딩 하나 켜면 #7·#8·#9가 동시에 진짜가 됨.
- 열린 점: 어떤 임베딩 모델/제공자, 비용·지연, 온프레미스 여부.

**C. (신뢰성 정책) LLM 안전장치를 어디까지 강하게?**
- 근거 검증: 지금은 **경고만**. → 환각 발견 시 자동 제거/재생성까지 갈지.
- Decision Memory: 추천이 기존 결정과 충돌할 때 **표시만** 함. → 자동 충돌 판정을 도입할지(현재는 LLM 근거검증 단계 몫으로 미룸).

**D. (인프라) 영속·실경로 검증**
- LangGraph 체크포인트를 영속화할지(정식 라이브러리 대기 vs 베타 감수).
- pgvector 실경로 벤치마크·ANN 인덱스·유사도 임계 튜닝을 언제 할지.

**E. (제품/범위) 남은 로드맵 우선순위**
- MCP(외부 Agent가 Clio 기능을 도구로 사용) 착수 시점.
- Web UI(백엔드 응답엔 근거·경고 등 다 있으나 화면 미노출).
- "관련 테스트 존재 여부"(위험도·난이도 판단에 중요한데 현재 약함) 보강.
- Issue Memory "학습 루프"(실제 수정 위치를 피드백 신호로 재활용) — 지금은 검색만.

## 4. 오늘 논의 안건 (우선순위순)

1. **[방향·최우선] rule-based 제거 & 에이전트형 전환** — 할 것인가, 언제, 어디까지? 이후 모든 우선순위가 여기서 갈림. (§3-A)
2. **[실효성] API 임베딩 활성화** — Memory 3종을 "진짜"로 만드는 스위치. 모델·비용 결정. (§3-B)
3. **[신뢰성] LLM 안전장치 강도** — 환각 검증을 경고→자동제거로, 결정 충돌 자동판정 도입 여부. (§3-C)
4. **[인프라] 영속 체크포인트 & pgvector 실경로 검증** — 언제·어떻게. (§3-D)
5. **[범위] 남은 로드맵 우선순위 합의** — MCP / Web UI / 테스트 탐색 / 점수화 보강 순서. (§3-E)

### 각 안건에서 받아야 할 결정
- 안건 1: "전환한다/안 한다", 한다면 목표 시점과 1차 범위.
- 안건 2: 임베딩 제공자·모델 확정, 비용 상한.
- 안건 3: 신뢰성 정책 수준(경고 유지 / 자동 개입).
- 안건 4: 영속화 착수 시점, 벤치마크 담당·기준.
- 안건 5: 다음 2~3개 작업 순서 확정.
